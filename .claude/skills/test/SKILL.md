# Test Runner

Run both test suites, report results, and fix any failures.

There are two suites:
- **Unit/API** (`npm test`) — Vitest, no server required, always run
- **E2E** (`npm run test:e2e`) — Playwright against `http://localhost:3000`, requires the app server + CouchDB to be running (via `docker compose`)

## Step 1 — Run the tests

Run both suites in sequence:

```bash
cd /mnt/Daten/projects/test/kanban/app && npm test 2>&1
```

```bash
cd /mnt/Daten/projects/test/kanban/app && npm run test:e2e 2>&1
```

## Step 2 — Report the results

Show a compact summary for each suite:
- Suite name, total / passed / failed / skipped count, duration
- If all tests in both suites pass: update MEMORY.md (see Step 2a) then confirm success and stop here
- If E2E fails because the server isn't reachable, note that clearly and don't treat it as a code bug

## Step 2a — Update test count in MEMORY.md (only when all tests pass)

Parse the Vitest passed count (the number after "Tests" in the summary line, e.g. `Tests  195 passed`) and the Playwright passed count (e.g. `7 passed`).

Update `/home/jean/.claude/projects/-mnt-Daten-projects-test-kanban/memory/MEMORY.md`:
- Find the lines that record Vitest and E2E test counts in the Test Suite section
- Update them if the counts have changed

Specifically:
- Look for `(NNN total)` patterns for Vitest counts and update to match
- Look for an E2E line (e.g. `### E2E (Playwright) — 7 tests`) and add or update it
- If counts already match, skip silently

## Step 3 — Analyze failures (only if tests failed)

For each failing test:

1. **Show the failure** — suite (Vitest/Playwright), test name, file path, error message, and the most relevant stack frame
2. **Read the source** — open the failing test file and the source file it exercises
3. **Diagnose the root cause** — distinguish between:
   - A real bug in the source code (fix the source)
   - A broken assertion that no longer matches the implementation (update the test)
   - A missing or changed API / function signature
   - An environment or setup issue (e.g. server not running for E2E)
4. **Propose a concrete fix** — show the exact change needed (diff or code block)

For Playwright failures, check `app/test-results/` for screenshots if available.

Group fixes by file. If multiple tests fail for the same root cause, explain that once rather than repeating.

## Step 4 — Offer to apply fixes

After presenting all fixes, ask: "Should I apply these fixes?"

- If yes: apply each fix with the Edit tool, then re-run the affected suite to confirm they pass
- If no: leave the code unchanged

## Notes

- Never apply fixes without the user's confirmation
- If a fix is uncertain, say so and present alternatives rather than guessing
- Prefer fixing source code over changing tests, unless the test expectation is genuinely wrong
- When re-running after a fix, show only the new summary (not the full output again)
- E2E tests live in `app/tests/e2e/` and use `app/playwright.config.js`
- The E2E auth state is cached in `app/tests/e2e/.auth-state.json` (regenerated by global-setup on each run)
